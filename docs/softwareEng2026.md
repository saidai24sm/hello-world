# ソフトウェア工学 2026

## ソフトウェア工学概論
ソフトウェアとは、「情報」を扱うプロダクトそのものや、プロダクトを提供する手段である。
具体的な定義としては、
- プログラムが適切に情報を扱うことを可能とするデータ構造
- プログラムの操作や使用法を記述した情報

以上の三要素を満たしたものがソフトウェアである。

また、特徴として、
- 新しい環境やニーズに適応し、ビジネス要求を実現するために強化されること
- より近代的なシステムやデータベースと相互運用するために拡張されること
- 進化する環境で実行可能であるように再構築されること

が求められる。

ソフトウェア工学におけるゴールは、  
**進化論に基づくソフトウェアが変化する状況に対応する方法論を考え出すことである。**  
具体的には、新しいソフトウェアが古いシステムから構築され、互換性を持ち、他システムと相互運用が可能であるような状態とすることである。

ソフトウェア開発において大切な原則としてHookerの7原則というものがある。以下がその内容である。
1. システムはユーザに価値を提供するためにある
2. 全ての設計は可能な限りシンプルであること
3. ビジョンを持ち続けること
4. 他の人が理解できるように開発を行うこと
5. 未来に対して可能な限り多くの選択肢を与えられるよう準備すること
6. 再利用を見越した計画を行うこと
7. 行動を起こす前に考えること

また、他にも覚えておくべきものとして、Hyrumの法則というものがある。
これは、**「あるAPIに十分数のユーザがいるとき、何を仕様として約束しているかに関わらず、システムの観察可能な挙動を仕様と考え、それに依存するコードを作成するユーザが発生する」**　というものであり、品質管理およびリリースのタイミングに深くかかわっている。

また、Hyrumの法則を深く理解することは、「変更すべきすべてのものが安全に変更可能で、かつコードベースの存続期間を通してそれが可能であるコード」であるソフトウェア工学のゴールに近い**スケーラブルなソフトウェア**の開発において重要である。

## プロジェクトに関して
プロジェクトとは、有期性、独自性を持つ類の仕事のことであり、ソフトウェア開発はこれに該当する。  
プロジェクトは目標に対し**バックキャスティング**という視点で進捗を確認し、作業を決定する。  
バックキャスティングとは、何を成し遂げたかという点を確認し、定めた目標に対し何が必要か計画し達成に向かうというものである。　　
目標は具体的かつ達成する価値のあるものとする必要があることは留意が必要である。
また、後述のソフトウェアの開発手法はバックキャスティングにより成り立っている。

## ソフトウェアの分析に関して
ソフトウェアの分析には様々なものがあり、物量（コード行数）、実行性能、開発工数、品質に対する分析などがある。  
しかし、品質、コスト、納期(QCD)のどの部分を重用しするかという点は重要である。  
また、先述のHyrumの法則を想定した品質管理、リリースが行われているかという点も重要である。

具体的な評価手法としては、ステップあたりのバグ発生率、FP法によるシステム規模の見積もりによる原価の推定、操作性や視認性などを考慮した使い勝手などがある。

# 開発手法に関して  

ソフトウェアの開発手法には様々なものがある。以下にそれを列挙する。
- ウォーターフォール型開発プロセス

上流工程から下流行程へ順に開発を行っていくというモデルであり、進捗管理が容易であり、成果物が明確であるというメリットがあるが、後工程にしわ寄せ発生するリスクがあるというデメリットがある。また、ウォーターフォール型開発プロセスの派生として、V字モデルというものがある。
- 　スパイラルモデル

プログラム開発を小さなフェーズに分割し、その中でプロトタイプ作成、提案、評価、フィードバックを繰り返し行うモデルである。プロトタイプ作成の業務量が大きくなるリスクがある。
- 反復型開発プロセス

ソフトウェアを機能分割し、これらを「反復」という単位で管理するものである。部分的な納品が可能である、要求に対して柔軟であるというメリットがあるが、全体像が見えずらいというデメリットがある。
- アジャイルプロセス

無駄を排し、最適な手法で動作するソフトウェアの提供を優先することで「できるだけ決定を遅らせる」「できるだけ早く提供する」を実現するプロセスである。人的リスクが大きいというデメリットがある。スクラム、カンバン、eXtreme Programmingなど様々な派生がある。

上記の開発プロセスにはそれぞれメリット、デメリット、リスクがあるため、それぞれの開発プロセスに関して理解を深め、適切なプロセスで開発を行うことが重要である。

# WBSに関して
WBSとは、プロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクトチームが実行する作業を、要素成果物を主体に階層的に要素分解したものであり、Work(作業) Brakedown(分解) Structure(構成図)の略である。  
これによりスコープおよびやるべき作業が明確となり、WBSに則り作業を行うのみでプロジェクトを完了できるため全体管理と作業化計画が明確化するというメリットがある。
